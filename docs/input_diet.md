# 입력 토큰 다이어트 (game-data trimming)

LIVE 스코어 챗봇 매트릭스에서 `fetch` 출력(특히 `match_info`)을 그대로 LLM에 넣으면 쿼리당 입력이 **~22k 토큰**에 달한다. 이 중 대부분은 챗봇 답변에 불필요한 raw 블록이다. 필요한 슬롯만 남기는 "다이어트"로 **비용·속도·grounding 정확도**가 모두 개선된다.

> **구현됨 (2026-06-09).** `uctest fetch --sport B --game-id <id> --diet` 로 적용한다. 변환 로직은 `uctest/data30/diet.py`(`apply_diet`/`diet_match_info`), 테스트는 `tests/test_diet.py`. 이 문서는 그 처방의 근거(측정·오독 원인)를 정리한다. (초기 검증은 jq/Python 임시 후처리로 했고, 그 처방을 코드화한 것이 현재 `--diet`다.)

## 1. 왜 필요한가 — 측정값

KBO 삼성 vs 두산 LIVE, 질문 7개 × Gemma-4 E4B(로컬 vLLM), system 비움, **순차 격리 측정**(큐 인플레이션 제거) 기준:

| 버전 | match_info 크기 | 평균 입력토큰 | 평균 latency | 스코어 정확 인식 |
|---|---|---|---|---|
| FULL (가공 안 함) | 30,588자 | 18,830 | 6,183ms | **0/7** (전부 오독) |
| DIET v1 (블록 제거) | ~5,000자 | 3,704 | 3,221ms | 2/7 |
| **DIET v2 (정규화)** | 3,175자 | **2,451** | **2,421ms** | **7/7** |

- **비용**: 입력 토큰이 전체 비용의 ~90%를 지배(출력은 수백 토큰뿐). 입력 7.7× 절감 → 비용도 그만큼.
- **속도**: prefill 감소로 로컬 latency 2.6× 단축.
- **정확도**: 노이즈 제거 + 스코어 정규화로 grounding 오독이 사라짐(아래 §3).

상세 근거: 메모리 `project_input_diet_and_latency.md`, 리포트 `testout/report-삼성-vs-두산-*-DIET-v2.html`.

## 2. 슬롯 분류 — 무엇을 남기고 무엇을 버리나

`fetch`(야구=DATA30 경로)의 `match_info` 하위 블록별 크기와 처리:

| 블록 | 크기(예) | 처리 | 이유 |
|---|---|---|---|
| `game_info` | ~0.4k자 | **유지**(현재 스코어 정규화에 사용) | 현재 스코어·이닝·상태의 진실원 |
| `betting_info` | ~0.8k자 | **유지** | 배당 질문 필수 |
| `live_board` | ~1.3k자 | **유지**(선택) | 볼카운트·주자 등 라이브 상황 |
| `player_info`(투수) | — | **요약 유지** | 선발 ERA 등. 단 §3-②처럼 라벨 분리 필수 |
| `player_info`(타자 전체 로스터) | ~11~16k자 | **요약 유지/조건부** | 타자 상세 질문 없으면 핵심만 |
| `team_rank`(리그 전체) | ~4.3k자 | **2팀만 유지** | 해당 두 팀 외 불필요 |
| `play_info`(투구별 시퀀스) | ~10k자 | **제거** | 챗봇 답변에 거의 불필요, 토큰 최대 소비원 |
| `vs.inning_info`(이닝별 누계) | — | **제거**(§3-① 필수) | 스코어 오독의 원인 |
| `recent_cheers` | ~1.1k자 | **유지** | 채널 분위기(이미 content만이라 lean) |

> 슬롯 선택은 **질문 유형에 따라 조건부**여야 한다. 예: "특정 타자 타율" 질문이면 타자 로스터를 남겨야 하고, 단순 스코어/배당 질문이면 더 쳐낼 수 있다. 에이전트 라우터가 `needed_slots`를 정하면 그게 곧 selective retrieval(=구조적 다이어트)이다.

## 3. v2 처방 — 오독을 잡은 두 가지 (★중요)

단순 블록 제거(v1)만으로는 스코어 오독이 2/7만 해결됐다. v2에서 아래 둘을 추가하니 **7/7 완전 해결**.

### ① 이닝 누계 raw 완전 제거 + 현재 스코어 단일 진실원

문제: `game_info.away_score/home_score`(현재 4:3)와 `vs.inning_info`/`scoreboard_record`(8회초까지 누계 3:0)가 **공존**하면, 모델이 후자를 최종 스코어로 오인한다. 사용자가 질문에 직접 준 스코어("3:4")조차 데이터 오독이 덮어쓴다.

처방:
- `vs.inning_info`·`scoreboard_record` 등 **이닝 누계 raw를 제거**.
- 현재 스코어를 **단일 필드로 명시**하고 "이 값만 쓰라"는 지시를 붙임:
  ```json
  "_현재스코어_유일정답": "두산베어스 2 : 1 삼성라이온즈 (3회말)",
  "_주의": "위 값이 현재 점수의 유일한 출처다. 이닝별 누계는 제공하지 않으니 반드시 이 값만 사용하고, 응원글에서 점수를 추측하지 말 것."
  ```

### ② 투수 시즌/금일 라벨 분리

문제: DATA30 투수 객체는 `innings_pitched`(시즌 누계)와 `pitch_count`(오늘)가 한 객체에 **혼재**한다. 모델이 "오늘 6이닝 25볼넷" 같이 **시즌 누계를 오늘 기록처럼 날조**한다(실측: e4b가 25WHIP 같은 왜곡 수치 생성, ERA 우위와 반대 해석).

처방: 투수 슬롯에 **신뢰 가능한 필드만** 남기고 라벨을 명확히:
```json
{ "선수명": "양창섭", "시즌ERA": "3.64", "시즌승패": "3승 0패",
  "오늘_투구수": "49",
  "주의": "이닝/실점/볼넷 누계는 시즌·금일 혼재로 신뢰 불가하여 제외. 오늘 세부 기록(이닝·실점)은 제공되지 않음." }
```
→ Q1 환각 사라지고 "오늘 세부 기록 없어 예측 어렵다"는 정직한 응답.

## 4. 운영 함정 (★실수에서 배운 것)

- **정규화 필드는 `user.jinja`가 렌더하는 슬롯 안에 넣어야 한다.** `match_info` **내부**에 넣어야 출력된다. 최상위 키(`current_situation` 등)로 두면 템플릿이 출력하지 않아 **통째로 누락** → 모델이 "스코어 정보 없음"이라 답함(v2 첫 시도가 이 함정으로 실패, `match_info` 안으로 옮겨 해결).
- `user.jinja` 슬롯 구조: `match_info` / `live_board` / `odds` / `history` / `recent_cheers` / `question`. 새 데이터는 이 중 하나에 실어야 한다.

## 5. 다이어트가 못 푸는 것 (다른 레버 필요)

- **history 누적**: 멀티턴이면 turn당 ~757토큰씩 history가 쌓인다. 다이어트는 게임데이터만 줄이지 history는 못 줄인다. DIET 기준 ~3턴째부터 history가 게임데이터를 추월 → **history 압축/요약**이 별도로 필요.
- **프롬프트 캐싱**: system·툴스키마 같은 고정 prefix는 캐싱으로 흡수(다이어트 영역 아님). 단 **라이브 스코어는 매 턴 변해 캐시가 안 되므로** 다이어트로만 줄일 수 있다. 다이어트와 캐싱은 상호보완(라이브 데이터 vs 정적 prefix).
- **베팅 권유 차단·자살 신호 안전 대응**: 데이터 처방으로 해결 안 됨. `system.jinja <policy>`로 명문화해야 함(no-system이면 전 모델 실패).

## 6. 구현 (`fetch --diet`)

`uctest/data30/diet.py`의 `apply_diet(fetched)`가 baseball fetch 출력을 받아 §2 슬롯 분류 + §3 정규화(현재 스코어 단일 진실원, 투수 라벨 분리)를 적용한다. `fetch --diet`가 `do_fetch(diet=True)`로 호출하며, `match_info`에 `game_info`가 없는(비-baseball/다른 스키마) 출력은 그대로 통과시킨다.

```sh
uctest fetch --sport B --game-id <id> --diet > /tmp/g_diet.json   # match_info ~22× 축소
```

남은 개선 여지:
- 슬롯 선택을 질문/intent 기반으로 조건부 처리하면 비용을 더 줄일 수 있다(에이전트 라우터의 `needed_slots` 연동). 현재 `--diet`는 고정 처방(전 질문 동일).
- §5의 history 압축·정책(system) 등 다이어트가 못 푸는 레버는 별도.
